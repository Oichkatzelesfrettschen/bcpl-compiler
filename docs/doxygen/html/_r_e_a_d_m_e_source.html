3This is an x86 (IA-32) and x86_64 with BIOS and UEFI (x64) port of the "classic" old BCPL compiler
4(around 1980) from the Tripos Research Group at Cambridge University. Experimental support exists for retro-modern x86 (IA-16) systems,
5for the compiler written in Chicken Scheme, written in Haskell, Written in Haskell orchestrating C via chicken scheme, and written in Go!
6
7BCPL was a popular systems programming language during the 1960s
8and 1970s, and is of great historical importance: about the time
9of the birth of UNIX, BCPL directly inspired the computer language
10B and thus had a very big influence on the development of C.
11
12The compiler available here is very close to that featured in the
13book, _BCPL: the language and its compiler_ by Martin Richards and
14Colin Whitby-Stevens (Cambridge: Cambridge University Press, 1979).
15
16As a real, working computer language implementation, that can be
17studied, modified, and played with, the classic BCPL compiler has
18a good deal to recommend it. The compiler frontend consists of
19only about 2000 lines of BCPL code, and (as supplied here) compiles
20to a static (fully-linked) x86 binary that is less than 36000 bytes
21in size.
22
23The present distribution supplies a compiler backend (OCODE to x86
24code generator), together with peephole optimizer, and reasonably
25extensive runtime support. A few revisions have been made to the
26compiler frontend -- it looks for header files in a standard location,
27for instance -- and the runtime incorporates support for UNIX
28command line arguments and error reporting. Some documentation
29that formed part of the original BCPL distribution tape is also
30included, as are a few utility programs.

32Requirements
33============
34
35The compiler relies on a standard set of development tools. Ensure at

37
38 clang
39 make

41 binutils

48Building
49========
50
51Before building you may install a comprehensive cross‑compilation
52environment by running ``./setup.sh`` as root. The script configures
53package repositories for multiple architectures and installs toolchains
54and emulators such as ``qemu`` to enable execution of foreign binaries,
55including 16‑bit and 32‑bit x86 programs. This is optional for a
56straightforward native build but recommended when targeting older or
57non‑native platforms.
58
59To build the compiler from the top-level directory, run::
60
61 ./scripts/makeall.sh
62 ./scripts/makeall.sh install
63
64The build now requires the Clang compiler with C23 support. The
65Makefile selects ``clang`` by default, so no extra arguments are
66needed. Use ``CC=gcc`` or another compiler only if it understands
67C23.
68
69Alternatively, you can build directly within ``src/``::
70
71 make -C src
72 make -C src install
73
74The Makefiles and CMake build files default to a 64-bit runtime.
75Specify ``BITS=32`` (or ``-DBITS=32`` when using CMake) for a 32-bit
76runtime or ``BITS=16`` for the experimental IA‑16 build. Cross builds
77may also set ``CROSS_PREFIX`` (for example ``CROSS_PREFIX=i686-linux-gnu-``
78or ``CROSS_PREFIX=ia16-elf-``) to select alternate assembler and linker
79commands. The BCPL word size (``WORDSZ``) is selected automatically from
80``BITS`` and will be 8 bytes for a 64-bit build, 4 bytes for 32 bit and
812 bytes for 16 bit.
82
83After installation, verify the compiler by running::
84
85 bcplc tools/cmpltest.bcpl
86
87By default the runtime is 64 bit, so running under ``qemu-i386`` is no
88longer required. A 32-bit build remains available for older
89environments via ``BITS=32`` or ``-DBITS=32``.

107Build
108
109See INSTALL for detailed build instructions. Build the compiler with
110
111 ./scripts/makeall.sh
112
113and then install it, optionally overriding `PREFIX` to choose an
114installation directory:
115
116 ./scripts/makeall.sh install
117
118The project also provides CMake files. A typical out-of-tree build
119looks like::
120
121 cmake -S . -B build -DCMAKE_INSTALL_PREFIX=/usr/local
122 cmake --build build
123 cmake --install build
124
125The CMake build regenerates ``st.s`` using the freshly built code
126generator by default. Set ``-DBOOTSTRAP=OFF`` to skip this step. The
127``tools`` and ``docs`` targets may be built from the same build
128directory.
129
130Ensure the resulting `bcplc` driver is on your `PATH` (or supply a
131`BC` variable pointing to it). If the driver has not yet been
132installed, the tests can be run with the freshly built copy in
133``src/``. The driver automatically searches its own directory for the
134runtime components, so no installation is required::
135
136 make -C tools BC=../src/bcplc test
137
138After installation the shorter
139
140 make test
141
142will also compile and execute the "cmpltest" suite using the
14364-bit runtime system.

155Experimental Go Port
156====================
157
158The `experiments/bcpl-go` directory contains an in-progress rewrite of
159selected components in Go. The module targets Go 1.24 and currently
160implements a basic version of the `bcplc` driver. Tests can be run with
161
162 go test ./experiments/bcpl-go
163
164The port is incomplete but provides a starting point for further
165translation and refactoring in Go.

167Experimental LLVM Backend
168=========================
169
170An optional tool named ``llcg`` can translate OCODE directly to LLVM IR.
171When LLVM is available, build it alongside ``cg`` and ``op`` via::
172
173 make -C src llcg
174
175or with CMake in an out-of-tree build::
176
177 cmake -S . -B build -DLLVM_DIR=/path/to/llvm/cmake
178 cmake --build build --target llcg
179
180The generated IR can be compiled to native code using ``clang`` or ``llc``.
181For example, to produce an object file directly::
182
183 ./llcg < input.O > prog.ll
184 clang -c prog.ll -o prog.o
185
186This bypasses the classic assembler pipeline used by ``cg`` and ``op``.
